# 专家证人助手

## 项目概述

面向法律诉讼场景，为专家证人（及代理律师）提供全流程AI辅助工具，解决海量证据材料处理、观点整理、庭前准备等痛点。

***

## 五大核心场景

### 场景一：争议差异智能分析

**需求描述**

- 从几百上千个证据文件中自动识别与争议焦点相关的材料
- 列出各方主张的差异点
- 标注支持各自主张的关键证据

**技术实现思路**

- 文档向量化存储（RAG架构）
- 争议焦点提取 → 证据关联检索
- 差异点对比表格生成

**输入**

- 争议焦点清单（或从起诉状/答辩状自动提取）
- 证据材料包（PDF/Word/图片等）

**输出**

- 争议差异对照表
- 每方证据支持度评分
- 缺失证据提醒

***

### 场景二：双方观点整理与追踪

**需求描述**

- 记录诉讼过程中各方（原告/被告/第三方）提出的观点、意见、主张
- 按时间线或争议点组织
- 识别观点变化与矛盾

**技术实现思路**

- 观点提取NLP模型
- 时间线可视化
- 矛盾点自动标记

**输入**

- 庭审笔录
- 书面意见
- 专家报告
- 往来函件

**输出**

- 观点时间线
- 各方立场对比图
- 矛盾点预警清单

***

**需求描述**

- AI扮演对方律师/法官进行质询
- 模拟庭审流程
- 评估专家证人回答的合理性与说服力

**技术实现思路**

- 角色扮演Agent（对方律师、法官、己方律师）
- 庭审流程引擎
- 回答质量评估模型

**输入**

- 专家证人背景资料
- 案件材料
- 预设角色风格（激进/温和）

**输出**

- 模拟庭审录音/文字记录
- 薄弱环节分析报告
- 改进建议

### PDF转图片（Java实现）

#### 方案一：Apache PDFBox（推荐）

**Maven依赖**

```xml
<dependency>
    <groupId>org.apache.pdfbox</groupId>
    <artifactId>pdfbox</artifactId>
    <version>3.0.7</version>
</dependency>
```

**核心代码**

```java
import org.apache.pdfbox.pdmodel.PDDocument;
import org.apache.pdfbox.rendering.PDFRenderer;
import org.apache.pdfbox.rendering.ImageType;
import javax.imageio.ImageIO;
import java.awt.image.BufferedImage;
import java.io.File;

public class PDFToImageConverter {
    
    /**
     * PDF转图片
     * @param pdfFile PDF文件路径
     * @param outputDir 输出目录
     * @param dpi 分辨率（建议150-300，越高越清晰但文件越大）
     * @param format 图片格式：png/jpg
     */
    public static void convertPdfToImages(File pdfFile, File outputDir, 
                                          float dpi, String format) throws IOException {
        try (PDDocument document = PDDocument.load(pdfFile)) {
            PDFRenderer renderer = new PDFRenderer(document);
            int pageCount = document.getNumberOfPages();
            
            for (int page = 0; page < pageCount; page++) {
                // renderImageWithDPI: 按DPI渲染，控制清晰度
                BufferedImage image = renderer.renderImageWithDPI(page, dpi, ImageType.RGB);
                
                File outputFile = new File(outputDir, 
                    String.format("page_%03d.%s", page + 1, format));
                ImageIO.write(image, format, outputFile);
            }
        }
    }
}
```

**关键参数**

| 参数        | 说明          | 推荐值               |
| --------- | ----------- | ----------------- |
| DPI       | 每英寸点数，决定清晰度 | 150（平衡）/ 300（高清）  |
| ImageType | 图片类型        | RGB（彩色）/ GRAY（灰度） |
| 格式        | 输出格式        | PNG（无损）/ JPG（压缩）  |

**性能参考**

- 100页PDF，DPI=150：内存占用约120MB，耗时约10-30秒
- 支持多线程并发处理（建议线程数=CPU核心数×2）

### 本地向量存储方案（2026最佳实践）

#### 方案选型矩阵

| 方案 | 数据规模 | 性能 | 部署复杂度 | 适用场景 |
|------|---------|------|-----------|---------|
| **SQLite + sqlite-vec** | 百万级以下 | 中等 | 极低（单文件） | 单机应用、快速原型 |
| **ChromaDB** | 千万级以下 | 良好 | 低（嵌入式） | Python项目、RAG应用 |
| **Qdrant** | 亿级 | 优秀 | 中（Docker） | 生产环境、高并发 |
| **Milvus Lite** | 千万级 | 优秀 | 低（嵌入式） | 需要Milvus生态 |
| **JVector** | 亿级 | 优秀 | 低（Java库） | Java项目、高性能需求 |

#### 推荐方案一：SQLite + sqlite-vec（最简单）

**特点**
- 单文件存储，零依赖
- 核心代码不足1MB
- 支持HNSW向量索引
- 任何支持SQLite的环境都能运行

**Java集成**
```java
// 通过JDBC使用sqlite-vec
// 1. 加载sqlite-vec扩展
// 2. 使用SQL进行向量操作

// 建表
CREATE TABLE documents (
    id INTEGER PRIMARY KEY,
    content TEXT,
    embedding FLOAT[1536]  -- 向量维度
);

// 创建向量索引
CREATE INDEX idx_embedding ON documents USING vec(embedding);

// 向量检索
SELECT id, content, vec_distance(embedding, ?) as distance
FROM documents
ORDER BY distance
LIMIT 10;
```

**适用场景**：证据材料在百万页以内，追求极简部署

#### 推荐方案二：ChromaDB（Python生态）

**特点**
- 嵌入式数据库，无需独立服务
- 默认使用SQLite持久化
- 原生支持Embedding自动计算
- Python API极简洁

**核心代码**
```python
import chromadb

# 持久化模式
client = chromadb.PersistentClient(path="./chroma_data")

# 创建集合
collection = client.create_collection("evidence_docs")

# 添加文档（自动向量化）
collection.add(
    documents=["文档内容1", "文档内容2"],
    metadatas=[{"case": "case_001"}, {"case": "case_001"}],
    ids=["doc_1", "doc_2"]
)

# 检索
results = collection.query(
    query_texts=["争议焦点描述"],
    n_results=5
)
```

**适用场景**：Python技术栈、快速验证MVP

#### 推荐方案三：Qdrant（生产级）

**特点**
- 专用向量数据库，性能最优
- 支持混合检索（向量 + 关键词）
- REST API + gRPC接口
- 有Dashboard可视化

**部署方式**
```yaml
# docker-compose.yml
version: '3.8'
services:
  qdrant:
    image: qdrant/qdrant:latest
    ports:
      - "6333:6333"
    volumes:
      - ./qdrant_storage:/qdrant/storage
```

**Java客户端**
```java
// Maven: io.qdrant:client
QdrantClient client = new QdrantClient(
    QdrantGrpcClient.newBuilder("localhost", 6334, false).build()
);

// 创建集合
client.createCollectionAsync("evidence", 
    Collections.VectorParams.newBuilder()
        .setSize(1536)
        .setDistance(Collections.Distance.Cosine)
        .build()
).get();
```

**适用场景**：生产环境、高并发、需要混合检索

#### 推荐方案四：JVector（Java原生高性能）

**特点**
- DataStax开源，Cassandra向量索引核心
- 纯Java实现，无JNI依赖
- 支持DiskANN + HNSW混合算法
- 磁盘感知设计，内存友好

**Maven依赖**
```xml
<dependency>
    <groupId>io.github.jvector</groupId>
    <artifactId>jvector-base</artifactId>
    <version>1.0.0</version>
</dependency>
```

**适用场景**：纯Java项目、超大规模数据、极致性能

#### 方案选择建议

| 你的情况 | 推荐方案 |
|---------|---------|
| 快速验证，Python技术栈 | ChromaDB |
| Java项目，百万级以下数据 | SQLite + sqlite-vec |
| Java项目，高性能要求 | JVector |
| 生产环境，团队运维能力强 | Qdrant |
| 需要全文搜索+向量混合 | Qdrant 或 Milvus |

### 相似度检索算法对比

不同向量数据库的相似度检索**不是同样的**，核心差异在三个层面：

#### 1. 距离度量（Distance Metric）

| 度量方式 | 公式特点 | 适用场景 | 支持的数据库 |
|---------|---------|---------|-------------|
| **余弦相似度 (Cosine)** | 计算向量夹角，忽略长度 | 文本语义检索（最常用） | 全部 |
| **欧氏距离 (Euclidean/L2)** | 直线距离 | 空间位置敏感的场景 | 全部 |
| **点积 (Dot Product)** | 考虑向量长度 | 推荐系统、需要权重区分 | Qdrant、Milvus |
| **MaxSim (Late Interaction)** | 多向量间的最大相似度 | 文档图片检索（ColQwen2） | 需特殊实现 |

**选择建议**：
- 文本/图片语义检索 → **Cosine**（最常用）
- 你的专家证人场景 → **Cosine** 即可

#### 2. 索引算法（Index Algorithm）

| 算法 | 原理 | 精度 | 速度 | 内存占用 | 代表数据库 |
|------|------|------|------|---------|-----------|
| **HNSW** | 分层导航小世界图 | 高 | 快 | 高 | Qdrant、Milvus、JVector |
| **IVF** | 倒排文件索引 | 中高 | 中等 | 中 | Milvus、Faiss |
| **DiskANN** | 磁盘感知图索引 | 高 | 中等 | 低（磁盘友好） | JVector |
| **FLAT** | 暴力扫描 | 100% | 慢 | 低 | 全部（小数据量） |

**选择建议**：
- 百万级以下 → **FLAT**（简单、精确）
- 百万级以上 → **HNSW**（平衡精度和速度）
- 超大规模 → **DiskANN**（省内存）

#### 3. 检索语法差异

**SQLite + sqlite-vec**
```sql
-- SQL风格
SELECT id, vec_distance(embedding, '[0.1, 0.2, ...]') as distance
FROM documents
ORDER BY distance
LIMIT 10;
```

**Qdrant**
```python
# REST API风格
client.search(
    collection_name="evidence",
    query_vector=[0.1, 0.2, ...],
    limit=10,
    with_payload=True
)
```

**JVector (Java)**
```java
// Java API风格
SearchResult result = index.search(
    queryVector, 
    10,  // topK
    Bits.ALL  // 过滤条件
);
```

#### 4. 关键差异总结

| 维度 | SQLite-vec | Qdrant | JVector |
|------|-----------|--------|---------|
| 默认距离度量 | Cosine | Cosine | Cosine |
| 索引算法 | HNSW | HNSW | DiskANN + HNSW |
| 是否支持过滤检索 | ✅ SQL WHERE | ✅  must/filter | ✅ Bits过滤 |
| 是否支持混合检索 | ❌ | ✅ 向量+关键词 | ❌ |
| 批量查询 | ❌ | ✅ | ✅ |

**重要提醒**：
- 同一批向量，用不同距离度量检索，结果会不同
- 建库时指定的距离度量，查询时必须一致
- 多模态模型（如ColQwen2）使用MaxSim，需要特殊处理

### 多模态向量化方案

#### 方案一：多模态嵌入模型（本地部署）

**核心思路**：不提取文字，直接将PDF页面截图送入多模态嵌入模型生成向量

**技术实现**：

- 使用 **ColQwen2/ColPali** 模型（开源，Apache 2.0许可）
- 每页PDF转为图片 → 多向量嵌入 → 向量数据库存储
- 查询时文本问题也编码为同一向量空间 → MaxSim相似度检索

**优势**：

- 无需OCR，避免文字提取失真
- 保留版面、图表、字体大小等视觉语义
- 对扫描件、复杂排版效果尤其好

**开源实现参考**：

- Weaviate官方recipe已提供完整代码
- 支持本地部署（M2 Pro Mac或T4 GPU即可运行）

#### 方案二：阿里云百炼多模态向量化（云服务）

**可用模型**：

| 模型                             | 维度       | 支持模态     | 单价(图片)         | 特点          |
| ------------------------------ | -------- | -------- | -------------- | ----------- |
| `qwen3-vl-embedding`           | 256-2048 | 文本+图片+视频 | 0.0018元/千token | 最新最强，支持融合向量 |
| `qwen2.5-vl-embedding`         | 512-2048 | 文本+图片    | -              | 融合向量专用      |
| `tongyi-embedding-vision-plus` | 1152     | 文本+图片+视频 | 0.0005元/千token | 独立向量，适合以图搜图 |
| `multimodal-embedding-v1`      | 1024     | 文本+图片+视频 | 0.0009元/千token | 稳定成熟        |

**计费说明**：

- 新用户免费额度：100万token（180天有效）
- 单张图片统一按128 tokens计算
- 示例：1000张PDF页面 ≈ 128,000 tokens ≈ 0.23元（用vision-plus）

**向量存储**：

- 阿里云DashVector（托管向量数据库）
- 或Milvus版向量检索服务
- 或自建Weaviate/PGVector

#### 方案对比

| 维度   | ColQwen2开源方案          | 阿里云百炼方案     |
| ---- | --------------------- | ----------- |
| 部署方式 | 本地/私有云                | 云服务API      |
| 数据隐私 | 完全本地，最安全              | 需上传图片到云端    |
| 成本   | 硬件成本                  | 按量付费，低成本    |
| 精度   | 高（late-interaction机制） | 高（通义多模态大模型） |
| 适用场景 | 敏感案件、本地化要求            | 一般案件、快速验证   |
| 技术门槛 | 需自建向量库                | 开箱即用        |

## 扩展：让LLM像理解代码一样理解案件

### 问题本质
律师/专家看图片没问题，但LLM需要**结构化表示**才能进行深度推理。就像代码有AST（抽象语法树），案件也需要"案件知识图谱"。

### 方案：GraphRAG（知识图谱+LLM）

**核心思路**
- 把案件资料转化为**知识图谱**（实体-关系-实体）
- LLM基于图谱进行**多跳推理**，而非片段检索
- 像IDE理解代码项目一样理解案件全貌

**案件知识图谱示例**
```
(原告:当事人) -[主张]-> (违约金:诉求)
(违约金:诉求) -[金额]-> (100万元:数值)
(违约金:诉求) -[依据]-> (合同第8条:条款)
(合同第8条:条款) -[约定]-> (延迟交付:条件)
(被告:当事人) -[抗辩]-> (不可抗力:理由)
(不可抗力:理由) -[证据]-> (气象证明:文件)
```

### 技术实现路径

#### 阶段1：案件结构化提取（从PDF到图谱）

**实体识别**
| 实体类型 | 示例 | 提取方式 |
|---------|------|---------|
| 当事人 | 原告、被告、第三人 | LLM NER |
| 时间 | 签约日期、交付日期 | 规则+LLM |
| 金额 | 合同金额、违约金 | 正则+LLM |
| 条款 | 合同条款、法律条文 | LLM抽取 |
| 证据 | 合同文本、邮件记录 | 文档关联 |
| 诉求 | 赔偿、解除合同 | LLM分类 |

**关系抽取**
```java
// 使用LLM从文本中提取三元组
Prompt: "从以下文本中提取实体关系三元组（主语-谓语-宾语）：
        原告张三主张被告李四违约，要求支付违约金100万元。"
        
输出: [
    ("张三", "是", "原告"),
    ("李四", "是", "被告"),
    ("张三", "主张", "违约"),
    ("张三", "要求", "违约金"),
    ("违约金", "金额", "100万元")
]
```

#### 阶段2：图谱存储与查询

**存储方案对比**

| 方案 | 适用规模 | 特点 | 推荐场景 |
|------|---------|------|---------|
| **Neo4j** | 亿级节点 | 成熟、Cypher查询、可视化 | 生产环境 |
| **NebulaGraph** | 十亿级 | 国产、分布式 | 大规模案件 |
| **Kuzu** | 百万级 | 嵌入式、单文件 | 本地应用 |
| **SQLite + 关系表** | 十万级 | 简单、零依赖 | MVP快速验证 |

**推荐：Kuzu（嵌入式图数据库）**
- 单文件存储（类似SQLite）
- 支持Cypher查询语言
- Java/Python绑定
- 适合本地案件分析

```cypher
// 创建实体
CREATE (p:Party {name: "张三", role: "原告"})
CREATE (c:Claim {type: "违约金", amount: 1000000})

// 创建关系
CREATE (p)-[:主张]->(c)

// 查询：某当事人的所有诉求
MATCH (p:Party {name: "张三"})-[:主张]->(c:Claim)
RETURN c.type, c.amount
```

#### 阶段3：GraphRAG检索与推理

**与传统RAG对比**

| 维度 | 传统RAG（向量检索） | GraphRAG（图谱检索） |
|------|-------------------|---------------------|
| 检索单元 | 文本片段 | 实体+关系子图 |
| 推理能力 | 单跳（直接匹配） | 多跳（链式推理） |
| 可解释性 | 低（黑盒相似度） | 高（可追溯关系链） |
| 适用问题 | "有哪些证据？" | "为什么这个证据支持这个主张？" |
| 幻觉风险 | 中（片段拼接） | 低（基于结构化事实） |

**GraphRAG查询示例**

```
用户问题："原告主张违约的依据是什么？"

传统RAG：
- 检索：返回包含"违约"的文本片段
- 问题：可能混杂被告的抗辩内容

GraphRAG：
1. 定位"原告"节点
2. 遍历：原告 → [主张] → 诉求 → [依据] → 条款/证据
3. 返回完整推理链：
   原告张三 → 主张违约金 → 依据合同第8条 → 约定延迟交付责任
4. LLM基于结构化路径生成答案
```

### 混合架构建议（向量+图谱）

```
┌─────────────────────────────────────────────────────────┐
│                     查询层                               │
│  用户问题 → 意图识别 → 路由分发                           │
├─────────────────────────────────────────────────────────┤
│                     检索层                               │
│  ┌─────────────┐    ┌─────────────┐                     │
│  │  向量检索    │    │  图谱检索    │                     │
│  │  (相似度)    │    │  (关系遍历)  │                     │
│  │             │    │             │                     │
│  │ "找相似证据" │    │ "找关联关系" │                     │
│  └─────────────┘    └─────────────┘                     │
├─────────────────────────────────────────────────────────┤
│                     数据层                               │
│  ┌─────────────┐    ┌─────────────┐                     │
│  │ SQLite-vec  │    │ Kuzu/Neo4j  │                     │
│  │  页面向量    │    │  案件图谱    │                     │
│  └─────────────┘    └─────────────┘                     │
└─────────────────────────────────────────────────────────┘
```

**使用场景分流**

| 用户意图 | 检索方式 | 示例 |
|---------|---------|------|
| 找相关证据 | 向量检索 | "与合同签署相关的材料" |
| 理清关系 | 图谱检索 | "原告的主张链条是什么" |
| 综合分析 | 混合检索 | "支持原告的证据有哪些" |

### 渐进式披露 + 人机协作流程

**核心原则**
> AI负责"找得到"，人负责"看得准"。不追求一步到位，而是通过多轮迭代逐步精化。

**三阶段工作流程**

```
┌─────────────────────────────────────────────────────────────┐
│  阶段1：AI粗筛（Recall优先）                                  │
│  ─────────────────────────                                   │
│  输入：争议焦点描述 + 全部PDF资料                             │
│  输出：Top-100 可能相关页面（按相似度排序）                    │
│  人工：快速浏览缩略图，勾选"相关/无关"（每页<3秒）            │
│  结果：人工剔除明显无关的资料，保留~30页                       │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  阶段2：AI精排（Precision优先）                               │
│  ─────────────────────────                                   │
│  输入：人工筛选后的30页 + 更具体的焦点描述                     │
│  输出：重新排序的Top-20 高相关页面                            │
│  人工：仔细阅读，标注关键段落，打标签（支持/反对/中立）         │
│  结果：形成证据清单，关联到争议焦点                           │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  阶段3：图谱构建（关系梳理）                                   │
│  ─────────────────────────                                   │
│  输入：精排后的证据 + 人工标注                                │
│  输出：案件知识图谱（当事人-诉求-证据关系）                    │
│  人工：审核图谱关系，补充隐性关联                             │
│  结果：完整的证据链，可用于庭审策略分析                        │
└─────────────────────────────────────────────────────────────┘
```

**为什么这样设计？**

| 阶段 | AI价值 | 人工价值 | 避免的问题 |
|------|--------|---------|-----------|
| 粗筛 | 从千页到百页，10倍压缩 | 快速判断相关性直觉 | AI幻觉导致漏检 |
| 精排 | 基于反馈重新排序 | 深度阅读理解 | 单纯依赖相似度的误判 |
| 图谱 | 自动提取显性关系 | 补充隐性逻辑 | 过度自动化导致的错误关联 |

**产品交互设计**

```
┌────────────────────────────────────────┐
│ 争议焦点：合同违约责任的认定           │
├────────────────────────────────────────┤
│                                        │
│  [阶段1/3] 粗筛结果（100页）            │
│  ┌────┐ ┌────┐ ┌────┐ ┌────┐          │
│  │ 📄 │ │ 📄 │ │ 📄 │ │ 📄 │  ...     │
│  │P15 │ │P23 │ │P47 │ │P52 │          │
│  │✅ │ │ ❌ │ │ ✅ │ │ ⏸️ │          │
│  └────┘ └────┘ └────┘ └────┘          │
│                                        │
│  [快速操作] 全选相关 | 全选无关 | 下一批 │
│                                        │
│  进度：已筛选 45/100 页，保留 12 页    │
│                                        │
└────────────────────────────────────────┘
```

**人工反馈循环**

```
人工筛选/标注
    ↓
积累高质量标注数据
    ↓
优化精排算法（基于规则的Re-rank）
    ↓
检索效率提升
    ↓
人工工作量减少
    ↓
可以处理更多案件
```

> 注：使用阿里云 `qwen3-vl-embedding` 模型，不对Embedding模型进行微调。
